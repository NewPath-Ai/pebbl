# Making pebbl work better — audit 2026-06-17

> **Historical record.** Several findings below concern `src/qmd.js` and QMD-based search. QMD (the `@tobilu/qmd` semantic search index) was removed from pebbl in M2; SQLite FTS5/BM25 is now the only search engine. Those QMD findings are obsolete. Kept as an audit record.

Code-grounded audit of the pebbl CLI (`~/factory/repos/pebbl`, v0.3.0). Method: 6 parallel audit dimensions read the real source, every finding was adversarially re-checked against the code by a separate skeptic, then synthesized. **42 findings raised, 42 confirmed, 0 refuted.** Generated from workflow wf_532df57f-4cf.

## The one thing to fix today
`context.js:222` prints `Run: pebbl narrative --refresh`. That flag does not exist; the token is swallowed as positional text and **overwrites the narrative body with the literal string `--refresh`**, silently, exit 0. It corrupted the sw-factory narrative twice (Jun 9, Jun 17). One-line hint change stops the active bleeding.

## Root-cause themes

### Parser fails open: unknown/malformed flags become content (root class)
src/args.js silently demotes any unrecognized --flag, --flag=value, -short token, or value-flag-with-no-value into positional text instead of erroring. Every content-writing command (narrative/log/handoff/feedback/search all do positional.join(' ')) then stores garbage. KNOWN_FLAGS is one global set, so handoff-only flags are 'known' everywhere and per-command typos slip through. This single design choice is the root cause behind the critical narrative corruption and a whole family of silent-data bugs. Value-flags also drop silently or eat the next word; numeric id-flags coerce to NaN/NULL with no error. The fix must be careful: context.js:420 deliberately relies on --full being demoted, and test/args.test.js:31-35 codifies pass-through as intended, so a blanket hard-reject breaks both.

### The narrative --refresh trap (parser bug x tool-suggested footgun, x6 duplicate)
context.js:222 prints 'Run: pebbl narrative --refresh' but --refresh was never implemented and narrative writeFileSync overwrites unconditionally with no .bak/confirm. Following the tool's own printed advice destroys the highest-visibility memory artifact, silently, with exit 0. Confirmed reproduced twice in prod (Jun 9, Jun 17). This is the convergence of the parser-fail-open class and a hint/command interface drift, landing on a destructive (overwrite, not append) command. It surfaced independently in 6 of the 6 audit dimensions, which is itself a signal of how central it is. Note the refusal guard must live in the narrative SET branch only, not inside writeNarrative (updateRefs reuses writeNarrative with legitimately short bodies).

### Handoff lifecycle is a single-slot illusion over an N-row model
Both the display query (context.js:81) and --close (handoff.js:51-53) use ORDER BY id DESC LIMIT 1, but the schema permits unlimited open rows and creation never closes priors. So newer opens bury older ones with no count, no staleness signal, no out-of-order close, and --list caps at 10 mixed-status rows. Close-discipline guidance only fires at read-time (context), never at create-time. The session auto-collect cutoff also keys off the last handoff ROW not the last close (a doc/code wording divergence, though the claimed double-count harm was disproven). And help/README still claim --close promotes to a foundation log, which the code no longer does (dead promoted_log_id branch).

### Recall is weaker than the docs promise (BM25 not semantic, scope = .pebbl only)
README/PEBBL sell 'semantic search, by intent' but src/qmd.js wires qmd's BM25 'search' subcommand and never runs 'qmd embed' — the vector index is empty, recall is literal token overlap. The collection root is .pebbl/ only, so prose decisions in sources/ and notes/ are structurally unreachable (the root of the worktree recall miss). --cat/--topic filter runs AFTER qmd's 5-result cap, so filtered search silently under-returns. The handoff --docs field is write-only (never materialized/indexed). The SQLite fallback (the real search when qmd is absent) does whole-query LIKE with no tokenization. qmd's degradation signals are swallowed. Implementer caveats: switching to 'qmd query' needs ANSI/progress-bar sanitization, a spawn timeout, and first-run model pre-fetch.

### Compaction nags forever but cannot deliver (grouping mismatch + manual-only)
The nag counts a flat per-topic population (no month, no category, no component_threshold, no corrected-entry exclusion) while the executor shatters the same entries across category/topic/MONTH buckets that must each independently clear threshold. So 245+33 entries fire the nag forever while preview rolls up nothing — the mechanical reason compaction 'never runs'. The month dimension makes long-lived topics permanently uncompactable. Compaction is 100% manual by design even though the mechanical rollup is safe (archive-first + transaction). Rollups also drop corrects edges, resurrecting superseded entries. Config drift compounds it: DEFAULT_CONFIG omits component_threshold and README documents wrong key names.

### The feedback->fix loop never closes; pebbl doesn't dogfood itself
Feedback is write-only: nothing in the default session path reads feedback.jsonl, it's undocumented in README/AGENTS/PEBBL, it has no resolved/triage state so it'd nag forever once surfaced, and outside a .pebbl tree it mints a stray .pebbl/ in cwd scattering reports. The proof is brutal: the narrative --refresh corruption bug sits in sw-factory/.pebbl/feedback.jsonl, captured and never actioned, which is exactly why it recurred. And the pebbl repo has no .pebbl/ (gitignored), so maintainers never experience the tool's own context output on a populated store — the fast feedback loop that would have caught all of the above never runs.

## All confirmed findings

### Critical (6)

**[M] Parser silently demotes unknown --flags to positional text (the flag-swallowing class of bug)** — `parser-fails-open-unknown-flags-become-content` (CLI arg parsing & input safety)
- evidence: src/args.js:22-25 — `if (!KNOWN_FLAGS.has(key)) { positional.push(arg); continue; }`. A mistyped/wrong --flag token is pushed verbatim into `positional`. Confirmed at runtime: `parseArgs(['--refresh'])` => `{flags:{}, positional:['--refresh']}`. This is even codified as intended in test/args.test.js:31-35 ('passes unknown flags through as positional' asserts positional===['--unknown','value']). Every positional-text command consumes that array as content: narrative.js:107 `positional.join(' ').trim()`, log.js:47, handoff.js:92, feedback.js:64, search.js:192.
- root cause: The parser was deliberately designed to fail open: any `--token` it doesn't recognize is treated as free text. That is safe for a flag-shaped substring *inside* a longer message (e.g. `pebbl log "run with --verbose"`) but catastrophic when the unknown flag is the entire intended invocation (e.g. `pebbl narrative --refresh`). The parser has no way to tell 'user meant a flag and typo'd it' from 'user meant literal text', and it never warns, so the wrong branch is silent. There is no validation layer above the parser either — bin/pebbl.js:48 dispatches raw args straight to each command.
- impact: Silent data corruption across the whole CLI. `pebbl narrative --refresh` (a string pebbl ITSELF prints, see context.js:222) overwrites the narrative body with the literal '--refresh'. This actually happened twice (Jun 9, Jun 17). The same trap exists for any near-miss flag on log/handoff/feedback: `--did` for `--done`, `--cats` for `--cat`, `--to-do` for `--todo` all silently become message content instead of doing what the user meant, with zero feedback.
- fix: Fix once at the parser (DRY) so every command is protected. Add a third return field `unknownFlags` and, more importantly, detect the dangerous case: a positional token that *starts with `--`*. In src/args.js, when `!KNOWN_FLAGS.has(key)`, still push to positional BUT record it in an `unknownFlags` array. Then add a tiny shared guard `assertNoStrayFlags(parsed, {command, knownFlagsForCmd})` called at the top of each command (or centralized in bin/pebbl.js before dispatch) that, if `unknownFlags.length > 0` AND the user clearly intended a flag (token is exactly `--word` with no spaces, not embedded in a sentence), prints `pebbl: unknown flag '--refresh' for 'narrative' (did you mean ...?). To store it as literal text, quote it.` and `process.exit(1)`. Default to refuse, not store. This converts a silent corruption into a loud, recoverable error.

**[S] feedback dropped outside a .pebbl tree silently creates a stray .pebbl/ in cwd, scattering reports** — `feedback-resolves-to-wrong-dir` (Feedback loop & dogfooding)
- evidence: resolvePebblDir() (feedback.js:14-20): if findPebblDir() returns nothing, it does `fs.mkdirSync(path.join(process.cwd(),'.pebbl'))` and writes there. So running `pebbl feedback` from a non-pebbl directory creates a brand-new .pebbl/ holding only feedback.jsonl. Unlike other commands which call requirePebblDir() (context.js:415, narrative.js:79) and refuse to run, feedback fabricates a directory.
- root cause: Feedback intentionally avoids requirePebblDir so a misbehaving install can still capture reports (feedback.js:11-13), but the fallback creates an orphan store with no init, no db, no AGENTS wiring — invisible to search and to any aggregation.
- impact: Feedback gets scattered into stray .pebbl/ dirs wherever the agent happened to be, fragmenting the very reports that already never get read. Reinforces the write-only failure: even the file location is unreliable.
- fix: When no .pebbl/ is found, write to a single known location (e.g. a user-global ~/.pebbl/feedback.jsonl) rather than minting a new project .pebbl/ in cwd, OR print a clear warning naming the created path and suggesting `pebbl init`. Pairs with the surfacing fix so a global feedback review can aggregate across repos.
- verifier correction: Two small inaccuracies in the finding, neither substantive: (1) the path is src/feedback.js, not feedback.js at repo root; (2) the mkdirSync is line 18 and uses { recursive: true } (the finding's paraphrase omitted the options arg). Also worth noting for scope: because findPebblDir walks UP the tree, the stray .pebbl/ is only minted when NO ancestor directory has a .pebbl/ — running feedback from a subdir of an existing project still resolves correctly. The finding's framing ("from a non-pebbl directory") is accurate for that case.

## Blind spots — what this audit did NOT cover (worth a follow-up)
- Concurrency / atomicity: every write is fs.writeFileSync / single better-sqlite3 calls with no file lock. Two agents (or the factory's concurrent /factory-work sessions noted in memory) running pebbl in the same .pebbl/ could interleave writes to narrative.md / handoffs.md / the db. Not examined — worth a follow-up given pebbl runs inside a multi-agent factory.
- Migration / upgrade safety: upgrade.js and a schema-migrate path exist (referenced in verdicts) but were not audited. The dead promoted_log_id column and any pending migrations could bite on version bumps; no test covers upgrade of a populated store.
- The qmd projection/materialization pipeline end-to-end: audit covered what gets indexed and how search reads, but not whether materializeHandoffsMd / manual-logs.md / commit-log.md stay consistent with the SQLite tables over time, nor what happens when qmd is a stale/mismatched version (the install is an external global).
- Test-suite coverage gaps as a root cause: several bugs are codified as intended by tests (args.test.js:31-35, 49-53; compact.test.js month-split). No audit of WHICH real behaviors lack tests — e.g. nothing asserts every flag a hint prints actually exists, nothing asserts context surfaces an open-handoff count. A 'hints reference real flags' contract test would prevent the whole interface-drift class.
- git-log / commit-log integration (log-commit.js): the handoff session cutoff reuses the same timestamp for git-commit collection (handoff.js:132); the commit-log path and its failure modes (empty repo, detached HEAD, large diffs) were not examined.
- Performance / scale: the unbounded-growth concern is named but no measurement — at what entry count does context's per-topic looping or the LIKE-scan fallback get slow? No profiling of the 245+33-entry real store.
- Security / input safety beyond flags: message/topic content is interpolated into markdown and shelled out to qmd via spawnSync arg arrays (safe-ish) but no review of path traversal in --docs paths, or of arbitrary content written to .md files that later feed an LLM context.
- The narrative refs/updateRefs correctness: updateRefs auto-remaps corrected refs and only re-stamps on corrections (not on plain new-foundation drift, per the regen finding) — its edge cases (a foundation entry deleted by compaction, a ref pointing at a rolled-up id) were noted but not fully traced.
